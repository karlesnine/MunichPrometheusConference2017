From Alexander Schwartz (.msg)

Talk about Start Your Engines: White Box Monitoring for Your Load Tests. And also about :

The RED method defines the three key metrics you should measure for every microservice in your architecture.

The USE method, this method is a methodology for analyzing the performance of any system. It directs the construction of a checklist, which for server analysis can be used for quickly identifying resource bottlenecks or errors. It begins by posing questions, and then seeks answers, instead of beginning with given metrics (partial answers) and trying to work backwards.

Lightning talk  / Various Links

  • Load testing: Gatling (scriptable, generate scala code, Akka based) vs JMeter (UI oriented, XML, threads)
  • Probe for  temperature, humidity, air pressure, and dewpoint in the Prometheus : Mouldy 
  • For Mysql I think you know the Mysqld Exporter. Since February it supports the implementation of Pt-heartbeat of percona for heartbeat.
  • Promagent is a Java agent providing Prometheus metrics for Java Web applications without modifying the application's source code.I'm not sure Arnauld agrees with this but it does not cost anything to look at
  • cloudflare/alertmanager2es Receives HTTP webhook notifications from AlertManager and inserts them into an Elasticsearch index for searching and analysis. Perfect for alert statistics
  • Zookeeper exporter zookeeper_exporter can be run in two modes: exhibitor and explicit.
  • Prom-metric-viewer A simple CLI tool to view prometheus metrics of exporters with style. It also includes a fancy Web UI to be even quicker viewing, sorting and searching metrics. With the multiplication of metrics this can become an interesting tool
  • Wizzy There can be a lot of use cases where wizzy can be used and some of them which I can think of right now are listed below. If you would like to do any of the following tasks in Grafana, then wizzy can help you accomplish them with its easy command line interface:
  1. Copy, move Grafana dashboards from one installation of Grafana to another. For example, from your dev installation to prod.
  2. Store Grafana dashboards, rows, panels, datasources, orgs and even template variables, etc. in Git.
  3. Copy, move, remove, edit rows and panels either within a dashboard or between two dashboards.
  4. Upload/Store/Download Grafana dashboards from and to AWS S3 respectively.
  5. Search/Download Grafana.net community dashboards in a single command.
  6. Generate 8 second GIFs for any Grafana dashboard with last 24 hours of data for your presentations.
  • PromHouse a long-term remote storage with built-in clustering and downsampling for Prometheus 2.0 on top of ClickHouse. Or, rather, it will be someday. Right now it's buggy and slow work-in-progress. Feel free to like, share, retweet, star and watch it, but do not use it in production.
  • Prometheus in Debian: 2 years on lightning talk about prometheus integration in Debian
  • Container Resource Consumption Estimates lightning talk from Michael Hausenblas (RedHat)
  • Unsee a Alert dashboard for Prometheus Alertmanager. Alertmanager UI is useful for browsing alerts and managing silences, but it's lacking as a dashboard tool - unsee aims to fill this gap. Starting with 0.7.0 release it can also aggregate alerts from multiple Alertmanager instances, running either in HA mode or separate. Duplicated alerts are deduplicated so only unique alerts are displayed. Each alert is tagged with names of all Alertmanager instances it was found at and can be filtered based on those tags.
